Signer Check register_user() · UserAccount
Contrato 1 — Defensa crítica
⚠

El Riesgo: Fake Signer Attack

Un atacante pasa una cuenta que él controla como si fuera tuya. Si la instrucción no verifica que el firmante de la transacción ES el dueño de la cuenta, puede registrar UserAccounts en nombre de otras wallets, secuestrar scores o acumular reputación fraudulenta.

Vulnerable — Sin PDA vinculada al signer
// ✗ PELIGROSO: user_account puede ser de cualquiera
#[derive(Accounts)]
pub struct RegisterUserUnsafe<'info> {
  #[account(mut)]                // ← sin seeds
  pub user_account: Account<'info, UserAccount>,
  pub signer: Signer<'info>, // ← no vinculado
}

// El atacante pasa user_account de VÍCTIMA
// y firma con SU propia wallet — Anchor lo acepta
// RESULTADO: reputación robada ✗
Seguro — PDA derivada del signer
// ✓ SEGURO: PDA vinculada matemáticamente al signer
#[derive(Accounts)]
pub struct RegisterUser<'info> {
  #[account(
    init, payer = signer,
    seeds = [b"user", signer.key().as_ref()],
    bump
  )]
  pub user_account: Account<'info, UserAccount>,
  pub signer: Signer<'info>, // ← firmante real
}

// Solo el dueño de la clave puede derivar esta PDA
// RESULTADO: imposible falsificar ✓
Flujo del ataque (sin defensa)
1

El atacante crea su propia wallet: Attacker.key()

2

Construye una tx que pasa victim_account como argumento de la instrucción

3

Firma con Attacker.key() — Anchor solo verifica que firmó, no que la cuenta es suya

4

La instrucción muta victim_account con datos del atacante — score robado

vs
Defensa con PDA vinculada
1

Anchor deriva la PDA: SHA256("user" + signer.key()) — solo existe una dirección válida por wallet

2

El atacante intenta pasar victim_account, pero su PDA no coincide con Attacker.key()

3

Anchor rechaza: ConstraintSeeds — la cuenta no pertenece al firmante

4

Transacción fallida. La víctima está protegida sin necesidad de código adicional

Derivación de la PDA — Inmutable y única
"user" + signer.key() → AiZB…Jp2 + program_id = PDA única e irrepetible

La PDA se calcula fuera de la curva elíptica Ed25519. No tiene clave privada — nadie puede firmar con ella. Solo el programa puede mover fondos o mutar datos de esta cuenta. Anchor verifica automáticamente que las seeds producen exactamente la dirección pasada.

Anchor Constraints en register_user()
✓ PASA
Signer<'info> — Verifica que la tx fue firmada criptográficamente por esta clave pública. Sin firma válida = transacción rechazada por el runtime de Solana.
✓ PASA
seeds = [b"user", signer.key()] — Deriva la PDA esperada y la compara con la cuenta pasada. Si no coincide, error ConstraintSeeds. Imposible pasar la cuenta de otra wallet.
✓ PASA
init — Solo se puede llamar una vez por PDA. Si la cuenta ya existe, error AccountAlreadyInitialized. Previene re-inicialización maliciosa.
✓ PASA
payer = signer — El firmante paga el rent de la cuenta. Garantiza que solo quien tiene SOL puede crear su propia cuenta — anti-spam económico.
✗ FALLA
Fake Signer Attack — El atacante pasa victim_account + firma con attacker_key. Anchor: PDA derivada de attacker_key ≠ victim_account → ConstraintSeeds → tx rechazada.

02 — VERIFIABLE BUILDS

Si no eres verificable, no eres de fiar

Un Verifiable Build permite que cualquier persona externa compile tu código fuente y verifique que el hash resultante es idéntico al programa desplegado en Solana. En el mundo DAO, es la diferencia entre confianza ciega y confianza matemáticamente demostrable.

Pendiente
Verificación del Programa
Program ID TU_PROGRAM_ID_AQUI
Hash compilado a1b2c3d4e5f6... (SHA256 del bytecode) ✓ Coincide
Hash on-chain a1b2c3d4e5f6... (verificado en Solana) ✓ Devnet
Cualquier persona puede compilar el código fuente desde programs/skillproof/src/lib.rs y comparar el hash con el programa en cadena.
1

Compilar con Docker determinista

Anchor usa un contenedor Docker con versión fija de compilador. Esto garantiza que el mismo código produce exactamente el mismo bytecode en cualquier máquina del mundo.

# Instalar la herramienta de verificación de Solana
cargo install solana-verify

# Compilar con entorno reproducible
anchor build --verifiable
Build complete. Verifiable: true
BPF hash: a1b2c3d4e5f6789abc...
2

Subir código fuente a un repositorio público

El código debe ser open source. Sin esto, nadie puede verificar nada. El commit exacto del despliegue queda vinculado al Program ID.

# En Anchor.toml — vincular repo y commit
[workspace]
members = ["programs/skillproof"]

[programs.devnet]
skillproof = "TU_PROGRAM_ID_AQUI"

# Verificar que el deploy usó el mismo commit
git log --oneline -1
a1b2c3d feat: SkillProof DAO v1.0 — Devnet deploy
3

Verificar contra el programa desplegado

Cualquier persona externa puede ejecutar este comando. Si los hashes coinciden, el código que ven en GitHub ES exactamente el código que se ejecuta en Solana.

# Verificar el programa on-chain desde cualquier máquina
solana-verify verify-from-repo \
  https://github.com/TU_ORG/skillproof-dao \
  --program-id TU_PROGRAM_ID \
  --library-name skillproof

✓ On-chain hash matches local build
✓ Program verified: TU_PROGRAM_ID
✓ Repository: github.com/TU_ORG/skillproof-dao
4

Registrar en OtterSec / Solana Verify

OtterSec mantiene un registro público de programas verificados en Solana. Una vez registrado, cualquier herramienta del ecosistema (exploradores, wallets) puede mostrar el badge de verificación junto a tu Program ID.

# Registrar en el verificador público de Solana
solana-verify upload-program \
  --program-id TU_PROGRAM_ID \
  --network devnet
